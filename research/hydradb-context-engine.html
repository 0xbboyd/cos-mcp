HydraDB Context Engine Architecture

Graph-backed lossless context compression plugin for Hermes Agent — HydraDB Cloud

Design Decisions

  • • Graph-backed compression vs lossy LLM summarization — no information loss (LCM pattern). Old messages are ingested as graph nodes; the agent queries the graph via tools when it needs compressed context.
  • • Cross-session persistence — the knowledge graph survives across sessions. on_session_start() reloads the graph; on_session_end() flushes final state. Same tenant_id: "hermes", per-profile sub_tenant_id.
  • • Shared SDK patterns — reuses _get_client() (lazy thread-safe singleton, double-checked locking), CircuitBreaker (5 failures → 120s cooldown), fire-and-forget daemon threads for ingest, and _formatter.format() for chunk extraction.
  • • Tool-augmented retrieval — the agent calls hydradb_context_search to query the graph and hydradb_context_expand to pull full context from compressed nodes, avoiding context-window pollution.
  • • Config-driven activation — selected via context.engine: "hydradb" in config.yaml. Only one context engine is active at a time.

Refs: cos_mcp/backends/hydradb.py L55-64 (_get_client), cos_mcp/circuit_breaker.py L56-61 (read trip, 5 failures → 120s)

ContextEngine ABC Contract

Required (abstract):

  • • name property → "hydradb"
  • • update_from_response(usage) — track token counts from API responses
  • • should_compress(prompt_tokens) → bool — gate compression decision
  • • compress(messages, current_tokens, focus_topic) → List[Dict] — ingest into HydraDB graph, return compressed message list with summary block

Optional hooks used:

  • • on_session_start(session_id, **kwargs) — load knowledge graph, provision tenant
  • • on_session_end(session_id, messages) — flush final graph state, shutdown client
  • • get_tool_schemas() → List[Dict] — expose hydradb_context_search, hydradb_context_expand
  • • handle_tool_call(name, args, **kwargs) → str — dispatch search/expand against HydraDB
  • • update_model(model, context_length, ...) — recalculate threshold from threshold_percent

Lifecycle order:

  • 1. Engine instantiated + registered (plugin register())
  • 2. on_session_start() — load graph
  • 3. Per-turn: update_from_response() → should_compress() → compress()
  • 4. on_session_end() — flush (CLI exit, /reset, gateway expiry)
  • 5. on_session_reset() — reset counters on /new or /reset

Integration with Memory Provider

  • • Shared client singleton — both HydraDBContextEngine and HydraDBMemoryProvider use the same HydraDBBackend._get_client() pattern (cos_mcp/backends/hydradb.py L55-64). A module-level _shared_backend reference avoids duplicate SDK connections.
  • • Same tenant / sub-tenant — tenant_id: "hermes", per-profile sub_tenant_id (auto-derived from profile name or set to "shared"). Context data lives alongside memory chunks but in distinct type namespaces (memory vs context).
  • • Circuit breaker reuse — the context engine instantiates its own CircuitBreaker (cos_mcp/circuit_breaker.py L16-87) with independent read/write gauges. Same 5-failure threshold, same 120s cooldown.
  • • Fire-and-forget ingest — both use daemon threads for non-blocking writes (threading.Thread(target=_run, daemon=True)). The context engine's compress() spawns an ingest thread and returns immediately with the summary block.
  • • graph_context=True — all HydraDB queries enable graph context for richer traversal results (backends/hydradb.py L74, L95; hydradb-memory/__init__.py L264). The context engine's search tool uses the same flag.
  • • Separate concerns — Memory Provider handles retrieval (what does the agent remember?). Context Engine handles compression (how do we fit everything in the context window?). Both coexist in the same plugin system.
