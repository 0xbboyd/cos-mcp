MuninnDB Context Engine — Architecture

Cognitive-native context compression plugin for Hermes Agent — ACT-R decay, Hebbian learning, Bayesian retrieval

Design Decisions — Cognitive-Native Compression

  • ACT-R temporal decay — Old context naturally loses activation; no manual pruning thresholds needed. Stale memories fade without explicit deletion or TTL management.
  • Hebbian co-activation learning — Frequently co-referenced facts strengthen their associative edges. The context graph self-organizes around the user's recurring topics and patterns.
  • Bayesian confidence gating — Retrieval returns quality-scored results. Contradicted or low-confidence memories are automatically suppressed, ensuring compressed summaries reflect the most reliable context.
  • 16 typed relationship types — Engine-native causes, contradicts, supports, depends_on, precedes, etc. capture richer inter-concept structure than flat text summaries or key-value stores.
  • Local, synchronous — MuninnDB runs on localhost:8475. No network latency means compress() can be fully synchronous — no daemon threading, no fire-and-forget. Engram storage completes before the compressed message list is returned.
  • Predictive Activation (PAS) — MuninnDB pre-activates context likely needed in the next turn, so retrieval is warm when compress() or tool calls query it.

ContextEngine ABC Contract

  • Required — Identity: name property returns "muninn". Selected via context.engine: "muninn" in config.yaml. Only one engine is active at a time.
  • Required — Token Tracking: update_from_response(usage) updates last_prompt_tokens, last_completion_tokens, last_total_tokens. Called after every LLM response. Must maintain compression_count.
  • Required — Compression Gate: should_compress(prompt_tokens) → bool. Returns True when compaction should fire. Default threshold: threshold_percent × context_length. Also respects protect_first_n and protect_last_n for head/tail preservation.
  • Required — Compaction: compress(messages, current_tokens, focus_topic) → List[Dict]. Main entry point. Receives full message list, returns compacted list. Muninn implementation: extract entities → store engrams in MuninnDB (ACT-R + Hebbian) → synthesize compressed summary block from high-confidence retrievals.
  • Optional — Session Lifecycle: on_session_start(sid, **kw) connects to MuninnDB and verifies vault health. on_session_end(sid, messages) flushes any pending engrams and closes the HTTP session. on_session_reset() zeroes counts.
  • Optional — Tools: get_tool_schemas() returns schemas for muninn_context_search and muninn_context_expand. handle_tool_call(name, args) dispatches to MuninnDB query or engram expansion.

Integration with MuninnDBMemoryProvider

  • Shared backend class — Both the Context Engine and Memory Provider use MuninnDBBackend from cos_mcp/backends/muninn.py. The Context Engine instantiates its own backend instance (separate session, same vault).
  • Same vault + config — Both read from ~/.hermes/muninn.json (base_url, vault) and MUNINN_API_KEY from ~/.hermes/.env. One vault per profile. Co-locating context and memory in the same vault lets Hebbian learning build cross-domain associations automatically.
  • Separate type tags — Memory provider tags engrams with "hermes-memory". Context engine tags with "hermes-context". Query filtering by tag isolates reads without schema changes — both providers share the vault but retrieve scoped results.
  • Independent circuit breakers — Each provider instance gets its own circuit breaker from cos_mcp.base_provider. Context engine failures don't trip the memory provider's breaker and vice versa. Default: 5 consecutive failures → 120s cooldown.
  • MAX_CONTENT_LENGTH = 15000 — Shared engram size limit from cos_mcp/backends/muninn.py. Context engine chunks large summaries to fit. Memory provider applies the same constraint for user-stored facts.
  • No delete support — MuninnDB's current API lacks delete-by-ID. The delete() method on MuninnDBBackend silently no-ops. Context engine relies on ACT-R decay to naturally age out stale context rather than explicit deletion.
